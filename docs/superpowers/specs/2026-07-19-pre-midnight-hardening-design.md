# P5 Demo 收尾硬化设计

## 目标

在不引入未经确认的小鹏私有车辆控制接口、且继续保持 `record_only=true` 与
`vehicle_command_transmitted=false` 的前提下，修复审查中已经复现的时序、证据和
界面状态问题，使比赛演示和后续实车接入的边界更清楚。

## 方案

### 后端与证据链

- `captured_at` 由服务端检查最大允许年龄和未来时钟偏差；未提供时间戳的旧版重试
  继续使用服务端接收时间，不改变既有幂等规则。
- 视频证据校验按后端相同的 canonical JSON 规则，从持久化 telemetry 重新计算
  `input_sha256`，而不是只比较证据文件里重复出现的字符串。
- 运行时继续只允许 `disabled` 或 `record-only`。无效、过期或已使用的授权令牌
  不得触发执行器；真正的车辆适配器需要后续单独设计“预留-执行-终结”事务，
  本次不伪造该流程。

### Web 端

- 所有本地参数编辑、场景切换、时间步进和恢复预设统一标记输入为 dirty，清除
  服务端事件/命令状态并提示重新提交。
- telemetry/authorization 的无效 API key 会退出 API 模式并清除持久化凭据；
  一次性车辆命令令牌失效只提示重试，不误删仍有效的连接。
- 连接请求采用候选代次校验；请求期间禁用凭据输入，旧响应不能覆盖新输入。

### Android 端

- XUI 调用异常或错误回调设置重连标记，成功连接后清除，避免告警灯/语音失败后
  永久停留在半连接状态。
- Activity 异步绑定记录请求状态，`onStop`、断开和绑定失败路径对称清理 callback，
  避免旧 Activity 被服务持有。
- API key 加密迁移不在本次范围内，待选择并验证 Android Keystore 依赖后单独处理。

## 验证标准

- 后端单元/API/evidence/benchmark 测试全部通过，并有过期、未来、篡改哈希和无效
  令牌不调用执行器的回归用例。
- Web `npm test` 全部通过，覆盖 dirty 状态、401 分类和连接竞态。
- Android JVM/Lint/现有构建测试通过；没有实车时不声称完成真实 XUI 验证。
- Release 文档明确写出 record-only 边界和下一阶段实车接入条件。

## 不选的方案

今晚不拆分车主身份、不实现外部签名、不启用真实 actuator，也不引入未经验证的
加密存储或两阶段执行协议；这些变化需要官方接口资料、设备测试和独立安全评审。
