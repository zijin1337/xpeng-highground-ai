# 评审对比、Benchmark 与能力边界

更新日期：2026-07-19

本文只比较各方案原本解决的问题，不把不同类别的能力包装成“全面优于”。当前仓库实现的是场端暴雨积水决策与留痕原型；车辆侧只有只读状态、语音和氛围灯提醒，车辆命令适配器默认为 `record-only`。车机工程使用 Open-Xpeng 社区整理的 P5 XUI SDK 类定义，不是小鹏官方开放的车辆控制接口。

## 结论先行

1. 小鹏智能泊车和记忆泊车主要解决“在已识别或已记忆的停车环境中，车辆如何完成低速路线辅助和泊入泊出”。
2. 高地 AI 解决的是另一个上游问题：接收场端积水证据，计算安全窗口，决定“现在是否仍允许迁移”，并记录决策、授权和命令证据。
3. 地磁、人工通知和传统防洪闸门都可复用。它们分别提供占用信息、人工兜底和物理防护，本方案补的是多源水情到逐车风险决策之间的闭环。

## 与现有方案对比

| 方案 | 原本解决的问题 | 能直接提供什么 | 暴雨积水场景中的缺口 | 与本方案的关系 |
|---|---|---|---|---|
| 地磁/车位检测 | 判断车位是否被占用、车辆是否存在 | 车位占用事件、停车时长等 | 不直接测量水深、上涨速度、路线是否干燥，也不计算最晚安全启动窗口 | 复用其占用事件作为车辆发现信号；仍由水位计、雨量计和场端状态提供风险证据 |
| 人工通知 | 由物业或车主判断并处置 | 电话、短信、群消息和人工确认 | 响应时间、触达确认、阈值一致性和审计完整性依赖人员 | 保留为 No-Go、通信异常和现场不确定时的兜底；系统提供统一证据和确认闭环 |
| 传统闸门/挡水板/排水设施 | 阻断、延缓或排出进水 | 场站级物理防护 | 不提供逐车水情、上涨窗口、干燥路线和车辆互锁判断 | 互补；闸门/泵状态应作为 `route_blocked` 或场端安全输入，而不是被替代 |
| 小鹏智能泊车/记忆泊车 | 在停车环境内辅助完成泊入、泊出和低速路线 | 车辆感知、路线记忆、调速、转向、避障等，具体能力依车型、市场和配置而异 | 本次检索的官方公开资料中未核验到接收地下车库场端水位、预测上涨窗口或在暴雨事件中主动跨车位迁移的能力 | 若未来取得正式授权接口，可作为经过独立安全验证的车辆执行层；当前仓库没有调用泊车控制能力 |
| 高地 AI 当前原型 | 判断暴雨积水条件下是否还有安全迁移窗口 | 多源遥测校验、确定性 Go/No-Go、一次性授权、SQLite 证据链、P5 提醒 | 场端传感器仍需部署；没有小鹏官方车辆控制接口；没有实车闭环验证 | 与上述方案组合，当前只做到风险决策、提醒和 `record-only` 留痕 |

## “AI 方案”的实际创新点

当前代码的安全核心是可解释的确定性决策引擎，不是黑盒机器学习模型。评审时应把创新点表述为“多源事件融合与可审计安全决策”，而不是泛称模型精度更高。

| 维度 | 单点告警/人工规则 | 当前原型 |
|---|---|---|
| 输入 | 单个水位阈值或人工观察 | 主辅水位、上涨速度、降雨、路线/出口、车辆健康、充电、乘员、定位、通信和人工接管状态 |
| 时间判断 | 到阈值后再通知 | 计算 `T_last = T_threshold - T_route - T_queue - T_buffer`，在窗口关闭前给出 `WATCH`、`PREPARE`、`MIGRATE_NOW` 或 `NO_GO` |
| 冲突处理 | 依赖人员解释 | 低置信度或主辅水位差过大时进入 `VERIFY_ONLY`，不下发移动权限 |
| 安全失败方式 | 可能继续使用旧通知 | 路线见水、互锁失败、事件过期或最新事件被替代时拒绝授权/命令 |
| 证据 | 通知记录分散 | 保存原始输入、SHA-256、决策、授权和命令记录 |
| 执行边界 | 容易把通知误写成自动控制 | 默认明确返回 `RECORDED_NOT_SENT`，没有制造商接口时不声称已控制车辆 |

后续若加入预测模型，适合让模型估计降雨到水位的时间序列或传感器异常概率；最终通行权限仍应受确定性安全闸约束。这样才能说明 AI 的增益，同时保留可验证的失效边界。

## 与小鹏泊车能力的公开证据矩阵

本节不是跨产品性能 Benchmark。当前没有在同一车辆、场地、传感器、网络和测试协议下采集小鹏泊车、本项目与涉水功能的实测数据，因此只做“公开资料是否支持该能力”的定性证据对齐，不比较速度、成功率或涉水深度。

### Demo 车型口径

固定场景采用 Open-Xpeng 社区资料标注的 `P5 / Xmart OS 3.6.1` 与社区 SDK `1.0.2` 作为车机兼容性假设。市场、年款、硬件配置和实车权限均为“未核验”，场景清单以结构化 `vehicle_profile` 保存这些未知项。它与官方公开的欧洲 XPILOT 2.5 版 P5 分开陈述，也不把国内部分激光雷达配置扩展成所有 P5 标配。

### 已核验事实

| 官方资料 | 已公开能力 | 对本项目的含义 |
|---|---|---|
| XPENG `XPILOT ASSIST` 页面 | 支持平行/垂直车位泊入和泊出，包含有线与无线车位；VPA 基于路线记忆，可辅助调速、转向、避障、避让来车和泊入 | 这些能力可回答停车环境中的“如何走、如何泊”；页面未描述场端积水接入和暴雨上涨窗口决策 |
| 同一页面的车型说明 | P5 被描述为首款量产搭载激光雷达并应用 XPILOT ASSIST 3.5 的车型；页面同时注明功能不适用于所有车型，部分功能依赖 OTA，应以配置表为准 | 不能把页面中的全部 XPILOT 3.5 功能写成所有 P5 标配 |
| XPENG 欧洲 P5 规格新闻 | 欧洲版 P5 使用 XPILOT 2.5；公开硬件为 5 个高清毫米波雷达、12 个超声波传感器、4 个行车摄像头和 9 个高感知摄像头；智能泊车在驾驶员监督下执行自动泊车 | 同名 P5 在市场、年份和配置上存在明显差异，Demo 必须标注所假定的车辆版本 |

### 能力边界对照

| 问题 | 小鹏公开泊车能力 | 高地 AI 当前原型 |
|---|---|---|
| 识别车位并完成泊入/泊出 | 公开资料确认支持，具体依车型/配置 | 不实现泊车轨迹规划或车辆控制 |
| 在停车场内按记忆路线低速辅助 | VPA 公开说明支持路线记忆和低速辅助 | 只接收 `route_dry`、`route_blocked` 等场端结果，不生成可执行轨迹 |
| 接入车库水位/雨量网关 | 本次官方公开资料检索未核验到 | HTTP 遥测契约已实现；真实传感器与网关待部署 |
| 预测水位到达禁行阈值的剩余时间 | 本次官方公开资料检索未核验到 | 已按水位、上涨速度、路线、排队和缓冲时间确定性计算 |
| 暴雨事件触发跨车位主动迁移 | 本次官方公开资料检索未核验到 | 当前只给出建议、单次授权和 `record-only` 命令留痕，未控制车辆 |
| 涉水通行 | 不应从电池包防护等级或泊车能力推导整车安全涉水深度 | 路线见水、水触发或达到禁行阈值即 `NO_GO`；目标是提前迁移，不是让车辆涉水 |

“未核验到”只表示下列公开官方页面没有提供该能力的证据，不等同于断言车辆或厂商内部系统永远不存在该能力。

### 可比性状态

| 比较维度 | 小鹏公开泊车能力 | 小鹏 P5 涉水能力 | 高地 AI 当前证据 | 是否可量化横比 |
|---|---|---|---|---|
| 测试对象 | 依市场/车型/配置的公开功能说明 | 本次公开资料未取得 P5 整车涉水原始限值 | 本地 FastAPI + 临时 SQLite + `record-only` | 否，对象不同 |
| 输入 | 车辆感知与驾驶员监督条件，完整协议未公开 | 未核验 | 固定 JSON 水情、路线和车辆互锁 | 否，输入不同 |
| 输出 | 泊入、泊出或记忆路线辅助 | 未核验 | 决策码、授权、证据与未发送命令留痕 | 否，任务不同 |
| 实测指标 | 本项目未实测 | 本项目未实测 | 合同正确性与进程内 API p50/p95 | 仅本系统内部可复现 |
| 车辆版本 | 官方页面提示依车型/配置；欧洲 P5 为 XPILOT 2.5 | 未核验 | P5 / Xmart OS 3.6.1 社区 SDK 假设；市场、年款、硬件未核验 | 否，版本未对齐 |

## 评审可直接使用的表述

> 小鹏泊车能力解决“车辆如何在停车环境中移动和泊入”；高地 AI 解决“暴雨事件下是否仍有安全移动窗口”。传统闸门负责减缓进水，地磁负责识别占用，人工负责异常兜底。我们的创新不是替代这些系统，而是把场端水情、车辆互锁、最晚安全窗口、一次性授权和审计证据串成一个可验证的逐车决策闭环。当前原型没有小鹏官方车辆控制接口，因此只做提醒和 `record-only` 留痕。
